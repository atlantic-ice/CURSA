## Тема: «Разработка программного комплекса для автоматизации проверки и оформления работ в соответствии с требованиями нормоконтроля»

# ВВЕДЕНИЕ

## Актуальность темы исследования

В современном образовательном процессе подготовка курсовых работ, дипломных проектов и выпускных квалификационных работ занимает значительное место в формировании профессиональных компетенций студентов. Одним из ключевых требований к таким работам является соблюдение единых стандартов оформления, устанавливаемых нормами нормоконтроля. Данные нормы включают требования к форматированию текста (шрифт, размер, межстрочный интервал), структуре документа (заголовки, нумерация разделов), оформлению библиографических ссылок, таблиц, рисунков и формул.

Традиционный подход к проверке документов предполагает ручной анализ соответствия требованиям нормоконтроля, что сопряжено с рядом существенных недостатков. Во-первых, данный процесс является крайне трудоёмким: проверка даже небольшого документа объёмом в 50 страниц может занимать от 30 минут до нескольких часов. Во-вторых, ручная проверка подвержена влиянию человеческого фактора — усталость, невнимательность и субъективность оценки приводят к пропуску ошибок и непоследовательности в их выявлении. В-третьих, при многократных правках документа возникает необходимость повторной проверки, что многократно увеличивает временные затраты.

Проблема особенно актуальна в условиях современного образования, где студенты выполняют несколько письменных работ одновременно, а преподаватели осуществляют проверку большого количества документов в ограниченные сроки. Существующие программные средства проверки документов, как правило, не предоставляют комплексного решения для автоматизации нормоконтроля, ограничиваясь отдельными аспектами проверки (например, только проверкой орфографии или только проверкой стилей).

В связи с этим разработка автоматизированной системы проверки документов на соответствие требованиям нормоконтроля является актуальной задачей, имеющей практическое значение для повышения качества образовательного процесса.

## Степень разработанности проблемы

Вопросам автоматизации проверки документов посвящен ряд научных работ и разработок. Существующие решения можно разделить на несколько категорий: текстовые процессоры с функциями проверки стилей (Microsoft Word, LibreOffice Writer), специализированные программы для проверки документов (Antijob, Справка-М), онлайн-сервисы для проверки текста на плагиат и орфографию, а также корпоративные системы управления документами.

Однако анализ существующих решений показывает, что большинство из них не обеспечивают комплексной проверки требований нормоконтроля в соответствии с ГОСТ 7.32-2017 и методическими указаниями конкретных учебных заведений. Отсутствует гибкая система настройки профилей проверки для различных ВУЗов и специальностей, а функции автоматического исправления ошибок реализованы ограниченно.

## Объект исследования

Объектом исследования является процесс проверки и оформления текстовых документов в соответствии с требованиями нормоконтроля.

## Предмет исследования

Предметом исследования являются методы и алгоритмы автоматизированной проверки документов формата DOCX на соответствие требованиям ГОСТ и внутренним стандартам учебных заведений.

# ГЛАВА 1. АНАЛИТИЧЕСКАЯ ЧАСТЬ

## 1.1. Анализ предметной области

### 1.1.1. Понятие нормоконтроля в образовательном процессе

Нормоконтроль представляет собой систему проверки документов на соответствие установленным нормам и правилам оформления. В контексте высшего образования нормоконтроль охватывает широкий спектр требований к письменным работам студентов, начиная от общих требований к форматированию текста и заканчивая специфическими правилами оформления библиографических ссылок.

Основными нормативными документами, регламентирующими требования к оформлению научных и учебных работ, являются:

- **ГОСТ 7.32-2017** — «Система стандартов по информации, библиотечному и издательскому делу. Отчёт о научно-исследовательской работе. Структура и правила оформления». Данный стандарт устанавливает общие требования к структуре и оформлению отчётов о НИР, однако его положения широко применяются и к другим видам учебных работ.
  
- **ГОСТ Р 7.0.5-2008** — «Система стандартов по информации, библиотечному и издательскому делу. Библиографическая ссылка. Общие требования и правила составления». Регламентирует правила оформления ссылок на использованные источники.

- **ГОСТ 7.1-2003** — «Система стандартов по информации, библиотечному и издательскому делу. Библиографическая запись. Библиографическое описание. Общие требования и правила составления». Определяет требования к библиографическому описанию источников.

Помимо государственных стандартов, каждое учебное заведение разрабатывает собственные методические указания, уточняющие и дополняющие требования ГОСТ с учётом специфики конкретных специальностей и направлений подготовки.

### 1.1.2. Типовые требования к оформлению документов

Анализ требований различных учебных заведений позволяет выделить следующие основные категории правил нормоконтроля:

**Требования к шрифтовому оформлению:**

| Параметр | Типовое значение | Примечание |
|----------|-----------------|------------|
| Основной шрифт | Times New Roman | Допускается Arial |
| Размер шрифта | 12-14 pt | Зависит от ВУЗ и типа работы |
| Начертание | Обычный | Полужирный только для заголовков |
| Цвет | Чёрный | Допускается для акцентов |

**Требования к параметрам страницы:**

| Параметр | Типовое значение |
|----------|-----------------|
| Левое поле | 30 мм |
| Правое поле | 15 мм |
| Верхнее поле | 20 мм |
| Нижнее поле | 20 мм |

**Требования к тексту:**

- Межстрочный интервал: 1,5
- Абзацный отступ: 1,25 см
- Выравнивание текста: по ширине
- Перенос слов: автоматический

**Требования к структуре документа:**

- Наличие обязательных разделов (титульный лист, введение, заключение, список источников)
- Иерархическая нумерация заголовков (1, 1.1, 1.1.1)
- Единообразие оформления заголовков одного уровня

### 1.1.3. Проблемы существующего подхода к нормоконтролю

Традиционный процесс нормоконтроля документов характеризуется рядом существенных проблем:

1. **Высокая трудоёмкость.** По экспертным оценкам, проверка документа объёмом 50 страниц занимает от 45 минут до 2 часов при условии тщательного анализа всех требований.

2. **Субъективность оценки.** Разные проверяющие могут по-разному оценивать степень соответствия документа требованиям, что приводит к непоследовательности результатов.

3. **Низкая скорость итераций.** При многократной правке документа студентом проверка каждой версии требует повторного анализа, что значительно увеличивает общее время на подготовку работы.

4. **Сложность для студентов.** Молодые исследователи не всегда знают все требования к оформлению, что приводит к систематическим ошибкам на начальных этапах подготовки документов.

5. **Нагрузка на преподавателей.** Проверка большого количества работ в сжатые сроки создаёт значительную нагрузку на преподавателей, отвлекая их от содержательной части рецензирования.

Решение указанных проблем возможно за счёт применения автоматизированных средств проверки документов.

## 1.2. Обзор существующих решений

### 1.2.1. Классификация существующих инструментов

Для анализа существующих решений в области проверки документов был проведён обзор доступных программных продуктов. Все найденные решения можно классифицировать по нескольким признакам.

**По типу реализации:**

- Встроенные функции текстовых процессоров (Microsoft Word, LibreOffice Writer, Google Docs)
- Специализированные настольные приложения
- Веб-сервисы (онлайн-проверка)
- Плагины и расширения для текстовых редакторов

**По функциональному назначению:**

- Проверка орфографии и грамматики
- Проверка стилистики и читаемости текста
- Проверка уникальности (антиплагиат)
- Проверка форматирования и соответствия шаблону
- Комплексные системы документооборота

### 1.2.2. Анализ аналогов

**Microsoft Word — встроенные средства проверки**

Microsoft Word предоставляет базовые инструменты для проверки форматирования: проверка стилей, поиск проблем макета, инспектор документа. Однако данные средства не обеспечивают комплексной проверки требований нормоконтроля.

*Преимущества:* широкая распространённость, интеграция с офисным пакетом, знакомый интерфейс.

*Недостатки:* ограниченные возможности проверки, отсутствие автоматического исправления, привязка к конкретному программному обеспечению.

**Справка-М**

Программа «Справка-М» разработана для проверки документов на соответствие методическим требованиям. Система предоставляет набор предустановленных профилей проверки для различных типов работ.

*Преимущества:* специализация на российских стандартах, наличие готовых профилей.

*Недостатки:* ограниченная масштабируемость, отсутствие веб-интерфейса, не поддерживает автоматическое исправление ошибок.

**Antijob**

Сервис проверки текста на плагиат и соответствие требованиям. Ориентирован преимущественно на выявление заимствований.

*Преимущества:* эффективное выявление плагиата, широкая база источников.

*Недостатки:* не предоставляет проверки форматирования, ограниченные возможности настройки профилей.

### 1.2.4. Выводы по результатам обзора

Проведённый анализ существующих решений позволяет сделать следующие выводы:

1. На рынке отсутствуют комплексные решения, обеспечивающие полный цикл проверки документов на соответствие требованиям нормоконтроля.

2. Большинство существующих инструментов не поддерживают функцию автоматического исправления обнаруженных ошибок.

3. Отсутствуют гибкие системы настройки профилей проверки для адаптации к требованиям конкретных учебных заведений.

5. Потребности в автоматизации нормоконтроля на российском рынке образовательных технологий остаются неудовлетворёнными.

Таким образом, разработка нового программного продукта, учитывающего выявленные недостатки существующих аналогов, является актуальной и востребованной задачей.

## 1.3. Постановка задачи

### 1.3.1. Формулирование целей и требований

На основании проведённого анализа предметной области и существующих решений формулируются следующие требования к разрабатываемому программному комплексу.

**Функциональные требования:**

1. Система должна обеспечивать загрузку документов в формате DOCX через веб-интерфейс.

2. Система должна выполнять проверку документа на соответствие требованиям нормоконтроля с использованием настраиваемых профилей.

3. Система должна предоставлять пользователю детализированный отчёт о найденных ошибках с указанием их типа, местоположения в документе и рекомендаций по исправлению.

4. Система должна поддерживать функцию автоматического исправления обнаруженных ошибок форматирования.

5. Система должна обеспечивать возможность управления профилями проверки (создание, редактирование, удаление).

6. Система должна поддерживать различные профили требований для разных учебных заведений и специальностей.

7. Система должна обеспечивать аутентификацию и авторизацию пользователей.

**Нефункциональные требования:**

1. Время проверки документа объёмом 50 страниц не должно превышать 30 секунд.

2. Система должна иметь веб-интерфейс, адаптированный для работы на устройствах с различными размерами экрана.

3. Система должна обеспечивать защиту пользовательских данных и загружаемых документов.

4. Система должна быть реализована с использованием современных технологий и обеспечивать возможность масштабирования.

### 1.3.2. Архитектурные решения

Для реализации сформулированных требований предлагается клиент-серверная архитектура системы, включающая следующие компоненты:

**Frontend (клиентская часть):**

- Веб-приложение на базе React
- Интерфейс загрузки документов
- Панель отображения результатов проверки
- Система управления профилями

**Backend (серверная часть):**

- API на базе Flask (Python)
- Подсистема парсинга документов DOCX
- Модуль валидации документов
- Модуль автокоррекции
- Система управления профилями
- Подсистема аутентификации

**Хранение данных:**

- Профили требований (JSON-файлы)
- Пользовательские данные (PostgreSQL/SQLite)
- Загруженные документы (файловая система)

**Backend:**

### 2.1.4. Взаимодействие компонентов

Взаимодействие между frontend и backend осуществляется посредством REST API по протоколу HTTP. Формат обмена данными — JSON.

Основные сценарии использования системы:

**Сценарий 1: Загрузка и проверка документа**

1. Пользователь выбирает файл DOCX и профиль проверки
2. Frontend отправляет файл и идентификатор профиля на backend
3. Backend сохраняет файл и инициирует проверку
4. Validation Engine последовательно запускает валидаторы
5. Результаты проверки сохраняются в БД
6. Frontend получает результаты и отображает отчёт

**Сценарий 2: Автокоррекция документа**

1. Пользователь нажимает кнопку «Исправить»
2. Frontend отправляет запрос на исправление
3. Backend запускает Correction Service
4. Для каждой исправляемой ошибки применяется соответствующий корректор
5. Исправленный документ сохраняется
6. Пользователю возвращается исправленный файл

## 2.4. Проектирование подсистемы валидации

### 2.4.1. Архитектура валидаторов

Подсистема валидации является ключевым компонентом программного комплекса, обеспечивающим анализ документа на соответствие требованиям нормоконтроля.

Валидаторы реализованы с использованием **шаблона проектирования «Стратегия»** (Strategy Pattern). Каждый валидатор реализует общий интерфейс `BaseValidator` и инкапсулирует логику проверки определённой группы требований.

```
│  + register_validator(validator)             │
│  + get_validators() → List[BaseValidator]    │
└────────────────────┬─────────────────────────┘
                     │ использует
     ┌────────────────┼────────────────┐
     ▼                ▼                ▼
┌─────────┐    ┌───────────┐    ┌─────────────┐
│  Font   │    │  Margin   │    │  Paragraph  │
│Validator│    │ Validator │    │  Validator  │
├─────────┤    ├───────────┤    ├─────────────┤
│+validate│    │ +validate │    │ +validate   │
└─────────┘    └───────────┘    └──────┬──────┘
     │                │                │
     └────────────────┼────────────────┘
                     │ наследуют
┌────────────────────┴─────────────────────┐
│              BaseValidator                 │
├───────────────────────────────────────────┤
│ + name: str                               │
│ + validate(doc, profile) → List[Issue]    │
│ # create_issue(...) → ValidationIssue     │
└───────────────────────────────────────────┘
```

### 2.4.2. Базовый класс валидатора

```python
    def validate(self, doc: Document, profile: dict) -> List[ValidationIssue]:
        """
        Выполнить проверку документа
        
        Args:
            doc: Объект документа python-docx
            profile: Словарь с параметрами профиля
            
        Returns:
            Список найденных проблем
        """
        pass
    
    def _create_issue(
        self,
        rule_id: str,
        severity: str,
        description: str,
        position: dict = None,
        can_autocorrect: bool = False,
        suggestion: str = None
    ) -> ValidationIssue:
        """Создать объект проблемы"""
        return ValidationIssue(
            validator=self.name,
            rule_id=rule_id,
            severity=severity,
            description=description,
            position=position or {},
            can_autocorrect=can_autocorrect,
            suggestion=suggestion
        )
```

### 2.4.5. Система оценки документа

Оценка соответствия документа требованиям рассчитывается по формуле:

```
Score = 100 × (1 - weighted_issues / max_possible_issues)

где:
- weighted_issues = Σ (issues[severity] × weight[severity])
- weight[critical] = 10
- weight[error] = 5
- weight[warning] = 2
- weight[info] = 1
- max_possible_issues = 100 (нормализующий коэффициент)
```

Шкала оценки:

- **90-100**: Документ соответствует требованиям
- **70-89**: Документ требует незначительных исправлений
- **50-69**: Документ требует существенных исправлений
- **< 50**: Документ требует значительной доработки

### 3.1.2. Реализация подсистемы валидации

Ядром серверной части является подсистема валидации, реализованная в файле `backend/app/services/validation_engine.py`. Данный модуль обеспечивает координацию работы всех валидаторов и формирование итогового результата проверки.

**Код Validation Engine:**

```python
from typing import List, Dict, Any
from dataclasses import dataclass, field
from docx import Document
from app.services.validators.base import BaseValidator, ValidationIssue


@dataclass
class ValidationResult:
    """Результат валидации документа"""
    passed: bool = True
    score: float = 100.0
    issues: List[ValidationIssue] = field(default_factory=list)
    validators_count: int = 0
    processing_time_ms: int = 0
    validator_results: Dict[str, List[ValidationIssue]] = field(default_factory=dict)


class ValidationEngine:
    """Движок валидации документов"""
    
    def __init__(self, validators: List[BaseValidator] = None):
        self.validators = validators or self._load_default_validators()
    
    def _load_default_validators(self) -> List[BaseValidator]:
        """Загрузка валидаторов по умолчанию"""
        from app.services.validators import (
            FontValidator, MarginValidator, ParagraphValidator,
            HeadingValidator, StructureValidator, BibliographyValidator,
            TableValidator, FormulaValidator, ImageValidator,
            AppendixValidator
        )
        return [
            FontValidator(),
            MarginValidator(),
            ParagraphValidator(),
            HeadingValidator(),
            StructureValidator(),
            BibliographyValidator(),
            TableValidator(),
            FormulaValidator(),
            ImageValidator(),
            AppendixValidator()
        ]
    
    def validate_document(
        self, 
        doc: Document, 
        profile: Dict[str, Any]
    ) -> ValidationResult:
        """Выполнить полную валидацию документа"""
        import time
        start_time = time.time()
        
        result = ValidationResult()
        result.validators_count = len(self.validators)
        
        for validator in self.validators:
            validator_name = validator.__class__.__name__
            
            # Получить параметры валидатора из профиля
            validator_config = profile.get(validator.type, {})
            
            if validator_config.get('enabled', True):
                issues = validator.validate(doc, validator_config)
                result.issues.extend(issues)
                result.validator_results[validator_name] = issues
        
        # Рассчитать оценку
        result.score = self._calculate_score(result.issues)
        result.passed = result.score >= 70.0
        
        # Подсчёт времени
        result.processing_time_ms = int((time.time() - start_time) * 1000)
        
        return result
    
    def _calculate_score(self, issues: List[ValidationIssue]) -> float:
        """Рассчитать оценку документа"""
        if not issues:
            return 100.0
        
        weights = {
            'critical': 10,
            'error': 5,
            'warning': 2,
            'info': 1
        }
        
        total_penalty = sum(weights.get(i.severity, 1) for i in issues)
        score = max(0.0, 100.0 - total_penalty)
        
        return round(score, 2)
```

### 3.1.3. Реализация FontValidator

Валидатор шрифтов проверяет соответствие текста документа требованиям к шрифтовому оформлению:

```python
from typing import List, Dict, Any
from docx import Document
from docx.text.paragraph import Paragraph
from docx.text.run import Run
from app.services.validators.base import BaseValidator, ValidationIssue, Severity


class FontValidator(BaseValidator):
    """Валидатор проверки шрифтов"""
    
    type = "font"
    
    def validate(
        self, 
        doc: Document, 
        profile: Dict[str, Any]
    ) -> List[ValidationIssue]:
        """Проверить шрифты в документе"""
        self.issues = []
        
        required_fonts = profile.get('allowed_fonts', ['Times New Roman'])
        required_size = profile.get('size', 14)
        size_tolerance = profile.get('size_tolerance', 0.5)
        
        for idx, paragraph in enumerate(doc.paragraphs):
            # Проверка каждого run (фрагмента текста с одинаковым форматированием)
            for run_idx, run in enumerate(paragraph.runs):
                if not run.text.strip():
                    continue
                
                self._check_font_name(
                    run, required_fonts, idx, run_idx
                )
                self._check_font_size(
                    run, required_size, size_tolerance, idx, run_idx
                )
        
        return self.issues
    
    def _check_font_name(
        self,
        run: Run,
        required_fonts: List[str],
        paragraph_idx: int,
        run_idx: int
    ) -> None:
        """Проверить имя шрифта"""
        font_name = run.font.name
        
        # Нормализация названия шрифта
        if font_name:
            font_name = font_name.strip()
        
        if font_name not in required_fonts:
            self.issues.append(self._create_issue(
                rule_id="font_name",
                severity=Severity.ERROR,
                description=f"Шрифт '{font_name}' не соответствует "
                           f"требованиям. Ожидается: {', '.join(required_fonts)}",
                position={
                    'paragraph': paragraph_idx,
                    'run': run_idx,
                    'text_preview': run.text[:50]
                },
                can_autocorrect=True,
                suggestion=f"Изменить шрифт на {required_fonts[0]}"
            ))
    
    def _check_font_size(
        self,
        run: Run,
        required_size: int,
        tolerance: float,
        paragraph_idx: int,
        run_idx: int
    ) -> None:
        """Проверить размер шрифта"""
        font_size = run.font.size
        
        if font_size is None:
            return
        
        # Размер в python-docx хранится в EMU (English Metric Units)
        # 1 point = 12700 EMU
        actual_size_pt = font_size.pt
        
        if abs(actual_size_pt - required_size) > tolerance:
            self.issues.append(self._create_issue(
                rule_id="font_size",
                severity=Severity.ERROR,
                description=f"Размер шрифта {actual_size_pt}pt не соответствует "
                           f"требуемому {required_size}pt",
                position={
                    'paragraph': paragraph_idx,
                    'run': run_idx
                },
                can_autocorrect=True,
                suggestion=f"Изменить размер шрифта на {required_size}pt"
            ))
```

### 3.1.4. Реализация подсистемы автокоррекции

Автокоррекция реализована в модуле `backend/app/services/correction_service.py`. Система выполняет многопроходное исправление ошибок с сохранением исходного содержания документа.

```python
from typing import List, Dict, Any, Optional
from docx import Document
from docx.shared import Pt, Cm
from docx.enum.text import WD_ALIGN_PARAGRAPH
from app.services.validators.base import ValidationIssue
from app.services.correctors import (
    FontCorrector, MarginCorrector, SpacingCorrector
)


class CorrectionService:
    """Сервис автоматического исправления ошибок"""
    
    def __init__(self):
        self.correctors = {
            'font_name': FontCorrector(),
            'font_size': FontCorrector(),
            'line_spacing': SpacingCorrector(),
            'paragraph_indent': SpacingCorrector(),
            'margins': MarginCorrector(),
        }
    
    def correct_document(
        self,
        doc: Document,
        issues: List[ValidationIssue],
        profile: Dict[str, Any]
    ) -> tuple[Document, List[Dict[str, Any]]]:
        """
        Выполнить исправление документа
        
        Returns:
            Tuple of (corrected_document, correction_log)
        """
        correction_log = []
        
        # Группируем исправления по типу для оптимизации
        issues_by_rule = self._group_issues_by_rule(issues)
        
        # Применяем исправления
        for rule_id, rule_issues in issues_by_rule.items():
            corrector = self.correctors.get(rule_id)
            
            if corrector and any(i.can_autocorrect for i in rule_issues):
                for issue in rule_issues:
                    if issue.can_autocorrect:
                        try:
                            corrector.correct(doc, issue, profile)
                            correction_log.append({
                                'rule_id': issue.rule_id,
                                'description': f"Исправлено: {issue.description}",
                                'status': 'success'
                            })
                        except Exception as e:
                            correction_log.append({
                                'rule_id': issue.rule_id,
                                'description': issue.description,
                                'status': 'failed',
                                'error': str(e)
                            })
        
        return doc, correction_log
    
    def _group_issues_by_rule(
        self, 
        issues: List[ValidationIssue]
    ) -> Dict[str, List[ValidationIssue]]:
        """Группировать проблемы по rule_id"""
        grouped = {}
        for issue in issues:
            if issue.rule_id not in grouped:
                grouped[issue.rule_id] = []
            grouped[issue.rule_id].append(issue)
        return grouped
```

```typescript
import axios, { AxiosInstance } from 'axios';
import type { 
  Profile, Document, ValidationResult, 
  UploadResponse, LoginRequest, LoginResponse 
} from '../types';

const API_BASE_URL = process.env.REACT_APP_API_URL || 'http://localhost:5000/api';

class ApiService {
  private client: AxiosInstance;

  constructor() {
    this.client = axios.create({
      baseURL: API_BASE_URL,
      timeout: 60000,
    });

    // Interceptor для добавления токена
    this.client.interceptors.request.use((config) => {
      const token = localStorage.getItem('access_token');
      if (token) {
        config.headers.Authorization = `Bearer ${token}`;
      }
      return config;
    });

    // Interceptor для обработки ошибок
    this.client.interceptors.response.use(
      (response) => response,
      async (error) => {
        if (error.response?.status === 401) {
          localStorage.removeItem('access_token');
          window.location.href = '/login';
        }
        return Promise.reject(error);
      }
    );
  }

  // Аутентификация
  async login(credentials: LoginRequest): Promise<LoginResponse> {
    const response = await this.client.post<LoginResponse>(
      '/v1/auth/login', 
      credentials
    );
    if (response.data.data?.access_token) {
      localStorage.setItem('access_token', response.data.data.access_token);
    }
    return response.data;
  }

  // Профили
  async getProfiles(): Promise<Profile[]> {
    const response = await this.client.get<{ success: boolean; data: Profile[] }>(
      '/v1/profiles'
    );
    return response.data.data;
  }

  // Загрузка документа
  async uploadDocument(
    file: File, 
    profileId: string
  ): Promise<UploadResponse> {
    const formData = new FormData();
    formData.append('file', file);
    formData.append('profile_id', profileId);

    const response = await this.client.post<UploadResponse>(
      '/v1/documents/upload',
      formData,
      {
        headers: { 'Content-Type': 'multipart/form-data' },
        onUploadProgress: (progressEvent) => {
          const percentCompleted = Math.round(
            (progressEvent.loaded * 100) / (progressEvent.total || 1)
          );
          console.log(`Upload: ${percentCompleted}%`);
        },
      }
    );
    return response.data;
  }

  // Результат проверки
  async getReport(documentId: string): Promise<ValidationResult> {
    const response = await this.client.get<{ 
      success: boolean; 
      data: ValidationResult 
    }>(`/v1/documents/${documentId}/report`);
    return response.data.data;
  }

  // Скачивание исправленного документа
  async downloadCorrected(documentId: string): Promise<Blob> {
    const response = await this.client.get(
      `/v1/documents/${documentId}/download`,
      { params: { corrected: true }, responseType: 'blob' }
    );
    return response.data;
  }
}
```
