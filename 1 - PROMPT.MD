# Ультимативный промпт для GitHub Copilot Agent

Ниже — полный промпт, который ты вставляешь в чат Copilot Agent. Он структурирован так, чтобы агент понимал контекст, архитектуру, каждую фичу и стандарт качества.

---

## 📋 Промпт (копируй целиком)

```
Ты — элитная команда из 50 senior-инженеров, UX-дизайнеров, экспертов по ГОСТ-стандартам и QA-специалистов. Твоя задача — довести до production-ready состояния моё приложение для автоматической проверки и исправления оформления академических работ (курсовые, дипломные, магистерские диссертации, рефераты, отчёты по практике) по стандартам ГОСТ.

## КОНТЕКСТ ПРОЕКТА

Приложение должно:
1. Принимать документы (.docx, .doc, .odt, .pdf — с OCR для PDF)
2. Анализировать ВСЁ оформление по ГОСТ и методическим рекомендациям вузов
3. Показывать ошибки с визуализацией прямо в документе
4. АВТОМАТИЧЕСКИ исправлять все найденные ошибки
5. Генерировать исправленный документ + детальный отчёт
6. Иметь современный, интуитивный UI профессионального уровня

## ТЕХНИЧЕСКИЙ СТЕК (если проект ещё не определён — используй этот)

### Backend:
- Python 3.12+ (FastAPI для API, async everywhere)
- python-docx для работы с .docx
- pdfplumber + pytesseract для PDF
- Celery + Redis для фоновых задач (обработка больших документов)
- SQLAlchemy + PostgreSQL для хранения истории проверок
- Pydantic v2 для валидации всех данных

### Frontend:
- React 18+ с TypeScript (strict mode)
- Vite для сборки
- TailwindCSS + shadcn/ui для UI компонентов
- Zustand для state management
- React Query (TanStack Query) для серверного состояния
- Framer Motion для анимаций
- react-pdf или custom renderer для предпросмотра документов

### Инфраструктура:
- Docker + docker-compose для развёртывания
- Nginx как reverse proxy
- Логирование через structlog
- Тесты: pytest + pytest-asyncio (backend), Vitest + Testing Library (frontend)
- CI/CD конфигурация

## ПОЛНЫЙ СПИСОК ПРОВЕРОК ПО ГОСТ (реализовать ВСЕ)

### 1. ШРИФТ И ТЕКСТ
- [ ] Шрифт основного текста: Times New Roman, 14 пт (настраиваемо)
- [ ] Шрифт в таблицах: допускается 12 пт
- [ ] Шрифт в подписях к рисункам: 14 пт
- [ ] Шрифт в сносках: 10-12 пт
- [ ] Цвет текста: только чёрный (проверка RGB/hex)
- [ ] Межстрочный интервал: 1.5 для основного текста
- [ ] Межстрочный интервал в таблицах: 1.0
- [ ] Выравнивание основного текста: по ширине
- [ ] Выравнивание заголовков: по центру (или по левому краю — настраиваемо)
- [ ] Абзацный отступ (красная строка): 1.25 см
- [ ] Запрет висячих строк
- [ ] Запрет разрыва абзацев между страницами (контроль)
- [ ] Автоматические переносы: включены/выключены (настраиваемо по вузу)

- [ ] Нижнее поле: 20 мм
```

````
## UI/UX ТРЕБОВАНИЯ (КРИТИЧЕСКИ ВАЖНО)

### Дизайн:
- Современный, минималистичный, профессиональный
- Тёмная и светлая тема (toggle)
- Цветовая схема: оттенки синего + акценты (зелёный = ОК, красный = ошибка, жёлтый = предупреждение)
- Скруглённые углы, мягкие тени, glassmorphism-элементы
- Типографика: Inter или Geist для интерфейса
- Адаптивность: desktop-first, но корректно на планшетах
- Микроанимации на все интерактивные элементы (hover, click, transitions)

### Главная страница:
- Большая drag & drop зона в центре с анимацией
- Текст: "Перетащите документ для проверки" + иконка
- Поддерживаемые форматы указаны
- Под зоной: краткая статистика (проверено документов, исправлено ошибок)
- Выбор шаблона (ГОСТ по умолчанию / конкретный вуз) перед загрузкой
- Анимация загрузки файла (прогресс-бар)

### Страница результатов проверки:
- LAYOUT: 3 колонки
  - Слева: список категорий ошибок с количеством (Шрифт (5), Поля (2), Заголовки (8)...)
  - Центр: превью документа с подсвеченными ошибками (красная рамка/фон для ошибок, жёлтая для предупреждений)
  - Справа: детали выбранной ошибки + кнопка "Исправить" / "Исправить все в категории"
- Верхняя панель: общий score (например, "Оформление: 67/100"), прогресс-бар, кнопки "Исправить всё", "Скачать отчёт", "Скачать исправленный документ"
- Фильтрация: по severity (ошибка/предупреждение/инфо), по категории
- Поиск по ошибкам
- При клике на ошибку — скролл к нужному месту в превью документа
- Сравнение До/После (split view)

### Анимации:
- Появление ошибок — fade in с задержкой (stagger animation)
- Score — анимированный счётчик
- Прогресс проверки — плавный progress bar с процентами и названием текущего этапа
- Skeleton loading для всех данных
- Confetti-анимация при 100/100 score 🎉
- Smooth scroll при навигации к ошибке

## ПАТТЕРНЫ КОДА И КАЧЕСТВО

### Backend:
- Все чекеры наследуются от AbstractChecker с методами check() и get_description()
- Каждый чекер возвращает List[Issue] с полями: severity, category, message, location, suggestion, auto_fixable
- FixerEngine принимает List[Issue] и документ, применяет исправления
- Все операции — async
- Comprehensive error handling (custom exceptions)
- Logging на каждом шаге
- Docstrings для каждой функции/класса
- Type hints ВЕЗДЕ
- 80%+ test coverage

### Frontend:
- Строгий TypeScript (no any, no ts-ignore)
- Все компоненты — функциональные с хуками
- React.memo для тяжёлых компонентов
- Proper error boundaries
- Accessible (ARIA labels, keyboard navigation)
- Оптимистичные обновления UI
- Debounce на поиск/фильтрацию
- Virtual scrolling для длинных списков ошибок
````

## API ENDPOINTS

```
POST   /api/v1/documents/upload          — Загрузка документа
GET    /api/v1/documents/{id}            — Получить информацию о документе
POST   /api/v1/documents/{id}/check      — Запустить проверку
GET    /api/v1/documents/{id}/check/status — Статус проверки (WebSocket тоже)
GET    /api/v1/documents/{id}/issues      — Список ошибок (с пагинацией, фильтрами)
POST   /api/v1/documents/{id}/fix         — Автоисправление (all или список issue_ids)
GET    /api/v1/documents/{id}/fixed       — Скачать исправленный документ
GET    /api/v1/documents/{id}/report      — Скачать отчёт (PDF/HTML)
GET    /api/v1/documents/{id}/compare     — Сравнение до/после

GET    /api/v1/templates                  — Список шаблонов
GET    /api/v1/templates/{id}             — Получить шаблон
POST   /api/v1/templates                  — Создать шаблон
PUT    /api/v1/templates/{id}             — Обновить шаблон
DELETE /api/v1/templates/{id}             — Удалить шаблон

GET    /api/v1/history                    — История проверок
GET    /api/v1/stats                      — Статистика
```

## ПРИОРИТЕТ РЕАЛИЗАЦИИ

Реализуй в следующем порядке:

1. Core: парсер документов + внутренняя модель документа
2. Core: система чекеров (начни с шрифтов, полей, интервалов — самые частые ошибки)
3. Core: система автоисправления
4. API: upload + check endpoints
5. Frontend: upload + результаты проверки (основной flow)
6. Остальные чекеры (заголовки, таблицы, рисунки, списки, ссылки)
7. Шаблоны правил + настройки
8. Отчёты
9. WebSocket + real-time прогресс
10. История + статистика
11. Docker + deployment
12. Тесты
13. Полировка UI/UX

## КРИТИЧЕСКИ ВАЖНО

1. НЕ делай заглушки и моки — реализуй РЕАЛЬНУЮ логику проверки
2. Каждый чекер должен РЕАЛЬНО парсить XML/OOXML структуру .docx через python-docx
3. Автоисправление должно РЕАЛЬНО модифицировать документ и сохранять валидный .docx
4. Код должен быть production-ready: error handling, logging, validation
5. Следуй SOLID, DRY, KISS принципам
6. Пиши самодокументируемый код с осмысленными именами
7. Каждый файл — не более 300 строк, декомпозируй
8. Проверяй edge cases: пустой документ, документ без стилей, corrupted файл, огромный документ

Начни с анализа текущей кодовой базы проекта, определи что уже реализовано, что нужно доработать, что переписать, и составь план действий. Затем начни реализацию по приоритетам.
